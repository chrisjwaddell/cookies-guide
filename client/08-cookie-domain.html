Attributes - domain

The domain attribute defines what domain the cookie should be sent to. If the domain is domain.com, the cookie will appear when requesting domain.com and www.domain.com. But if the domain is www.domain.com, the cookie won't appear for domain.com.

If a subdomain is given as a domain attribute for a cookie, eg api.domain.com, then the request must start with that subdomain and match on the domain too. So a request for www.domain.com won't match.

document.cookie = "all_domain=google.com;domain=google.com;"
This cookie appears on all this website sub-domains.

The following cookie is the same thing, it appears on all sub-domains .domain.com:
"all-domain=all; domain=.domain.com;"

The following cookie will show for only www.domain.com:
"just-wwww=www; domain=www.domain.com;"

The following cookie will show for domain.com:
"just-domain=domain; domain=domain.com;"

A cookie domain is set to herokuapp.com but rejected on serene-bastion-01422.herokuapp.com because it comes from a domain included in the Public Suffix List.

The Public Suffix List is a list maintained by Mozilla, used by all browsers to restrict who can set cookies on behalf of other domains.

See more at:
https://publicsuffix.org/
https://devcenter.heroku.com/articles/cookies-and-herokuapp-com

The browser has the following rules when it recieves a response:
• Reject the cookie altogether if either the domain or the subdomain in Domain don't match the sender host
• Reject the cookie if the value of Domain is included in the Public suffix list
• Accept the cookie if the domain or the subdomain in Domain matches the sender host
